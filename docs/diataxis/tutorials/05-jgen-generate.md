# Tutorial 5 — Scaffold a CRUD Service with jgen in One Command

> **Learning goal:** use the `jgen` code generator to produce a complete, compilable, testable CRUD service — including records, a `Result`-based service layer, and a JUnit 5 test suite — from a handful of CLI commands.

---

## Prerequisites

- Completed [Tutorial 4 — Railway Error Handling](04-result-railway.md)
- `jgen` is available: `bin/jgen list` should print the template catalogue
- If `jgen` is not installed, follow the [Install ggen](#appendix-install-ggen) appendix at the end of this tutorial

---

## What you will build

A **`UserService` CRUD layer** that manages `User` records, including:

| Artifact | Template used |
|---|---|
| `User.java` — immutable record with validation | `core/record` |
| `UserValidation.java` — `Result`-based validators | `error-handling/result-railway` |
| `UserService.java` — service layer (create, find, update, delete) | `patterns/service-layer` |
| `UserServiceTest.java` — JUnit 5 test suite | `testing/junit5` |

By the end of this tutorial you will know how to: list templates, generate individual artefacts, understand what was generated, and run the resulting tests.

---

## Background: what is jgen?

`jgen` wraps [ggen](https://github.com/seanchatmangpt/ggen) — a Rust-based code generator driven by **Tera templates** and **RDF ontologies**. Each template produces idiomatic Java 26 code for a specific pattern:

- Templates live in `templates/java/**/*.tera`
- The `schema/*.ttl` ontologies define Java type-system relationships
- `bin/jgen` is the CLI entry point

You never hand-write the repetitive boilerplate (record constructors, null checks, `Result` wrappers, test scaffolding). You describe *what* you want; `jgen` generates the *how*.

## Step 2 — Generate the User record

Generate the domain record first. The `-t` flag selects the template, `-n` sets the class name, and `-p` sets the package:

```bash
bin/jgen generate -t core/record -n User -p com.example.model
```

`jgen` writes the generated file to the appropriate path under `src/main/java/` and prints the path:

```
Generated: src/main/java/com/example/model/User.java
```

Open the file and read it. You will see something like:

```java
package com.example.model;

import java.util.Objects;

/**
 * Immutable User value object.
 *
 * <p>Generated by jgen from template core/record.
 * Customise the fields below then run {@code mvnd spotless:apply} to format.
 */
public record User(
        long id,
        String name,
        String email
) {
    /** Compact canonical constructor — validates non-null fields. */
    public User {
        Objects.requireNonNull(name,  "name must not be null");
        Objects.requireNonNull(email, "email must not be null");
    }

    /** Fluent builder for constructing User instances. */
    public static Builder builder() {
        return new Builder();
    }

    public static final class Builder {
        private long   id;
        private String name;
        private String email;

        public Builder id(long id)         { this.id = id;       return this; }
        public Builder name(String name)   { this.name = name;   return this; }
        public Builder email(String email) { this.email = email; return this; }

        public User build() {
            return new User(id, name, email);
        }
    }
}
```

The template has given you a record with:
- A compact canonical constructor with null validation
- A fluent builder
- Sensible field names (customise as needed)

**Customise the record** — add the fields your domain needs. For this tutorial, keep the generated `id`, `name`, and `email` fields. You could add `age` or `role` if you want to practise.

## Step 3 — Generate the validation layer

Now generate a `Result`-based validation class:

```bash
bin/jgen generate -t error-handling/result-railway -n UserValidation -p com.example
```

```
Generated: src/main/java/com/example/UserValidation.java
```

The generated file will contain:

- A sealed `UserValidationError` inner interface (or a top-level one, depending on the template version)
- Static `validateName(String)`, `validateEmail(String)` methods returning `Result<String, UserValidationError>`
- A `validate(User)` method that chains the individual validators with `flatMap`

Scan through the file. Notice that the generated validators mirror what you built by hand in Tutorial 4 — but this time `jgen` wrote the boilerplate in seconds.

**Do not be afraid to edit generated code.** Generation is a starting point, not a constraint. Add domain-specific rules (maximum name length, company email domain, etc.) directly in the generated file.

## Step 4 — Generate the service layer

```bash
bin/jgen generate -t patterns/service-layer -n UserService -p com.example.service
```

```
Generated: src/main/java/com/example/service/UserService.java
```

The service layer template generates a class with:

```java
package com.example.service;

import com.example.model.User;
import com.example.UserValidation;
import org.acme.Result;
import java.util.Map;
import java.util.Optional;
import java.util.concurrent.ConcurrentHashMap;
import java.util.concurrent.atomic.AtomicLong;

/**
 * CRUD service for User, using Result<T,E> for all operations.
 *
 * <p>Generated by jgen from template patterns/service-layer.
 */
public final class UserService {

    private final Map<Long, User> store     = new ConcurrentHashMap<>();
    private final AtomicLong      idCounter = new AtomicLong(1);

    public Result<User, UserValidation.UserValidationError> create(String name, String email) {
        return UserValidation.validateName(name)
                .flatMap(validName -> UserValidation.validateEmail(email)
                    .map(validEmail -> {
                        long id   = idCounter.getAndIncrement();
                        var  user = User.builder().id(id).name(validName).email(validEmail).build();
                        store.put(id, user);
                        return user;
                    }));
    }

    public Optional<User> findById(long id) {
        return Optional.ofNullable(store.get(id));
    }

    public Result<User, UserValidation.UserValidationError> update(long id, String name, String email) {
        return UserValidation.validateName(name)
                .flatMap(validName -> UserValidation.validateEmail(email)
                    .flatMap(validEmail -> {
                        if (!store.containsKey(id)) {
                            return Result.failure(new UserValidation.UserValidationError.NotFound(id));
                        }
                        var updated = User.builder().id(id).name(validName).email(validEmail).build();
                        store.put(id, updated);
                        return Result.success(updated);
                    }));
    }

    public boolean delete(long id) {
        return store.remove(id) != null;
    }

    public java.util.Collection<User> findAll() {
        return java.util.List.copyOf(store.values());
    }
}
```

Every mutating operation goes through validation. The `ConcurrentHashMap` makes the service safe for concurrent use — relevant since `Proc` runs on virtual threads.

## Step 5 — Generate the test suite

```bash
bin/jgen generate -t testing/junit5 -n UserService -p com.example.service
```

```
Generated: src/test/java/com/example/service/UserServiceTest.java
```

The generated test class is a full JUnit 5 suite:

```java
package com.example.service;

import com.example.model.User;
import org.assertj.core.api.WithAssertions;
import org.junit.jupiter.api.BeforeEach;
import org.junit.jupiter.api.DisplayName;
import org.junit.jupiter.api.Nested;
import org.junit.jupiter.api.Test;

@DisplayName("UserService")
class UserServiceTest implements WithAssertions {

    private UserService service;

    @BeforeEach
    void setUp() {
        service = new UserService();
    }

    @Nested
    @DisplayName("create()")
    class Create {

        @Test
        @DisplayName("returns a User when input is valid")
        void validInputCreatesUser() {
            var result = service.create("Alice", "alice@example.com");

            assertThat(result.isSuccess()).isTrue();
            result.fold(
                user -> {
                    assertThat(user.name()).isEqualTo("alice");   // normalised
                    assertThat(user.email()).isEqualTo("alice@example.com");
                    assertThat(user.id()).isPositive();
                    return null;
                },
                err -> { fail("Unexpected error: " + err); return null; }
            );
        }

        @Test
        @DisplayName("rejects an invalid email")
        void invalidEmailReturnsFailure() {
            var result = service.create("Bob", "not-an-email");

            assertThat(result.isFailure()).isTrue();
        }

        @Test
        @DisplayName("rejects a blank name")
        void blankNameReturnsFailure() {
            var result = service.create("", "valid@email.com");

            assertThat(result.isFailure()).isTrue();
        }
    }

    @Nested
    @DisplayName("findById()")
    class FindById {

        @Test
        @DisplayName("finds a user that was created")
        void findsExistingUser() {
            var user = service.create("Carol", "carol@example.com").orElseThrow();
            var found = service.findById(user.id());

            assertThat(found).isPresent().get()
                    .extracting(User::name)
                    .isEqualTo("carol");
        }

        @Test
        @DisplayName("returns empty for unknown id")
        void returnsEmptyForUnknownId() {
            assertThat(service.findById(9999L)).isEmpty();
        }
    }

    @Nested
    @DisplayName("update()")
    class Update {

        @Test
        @DisplayName("updates an existing user successfully")
        void updatesExistingUser() {
            var user    = service.create("Dave", "dave@example.com").orElseThrow();
            var updated = service.update(user.id(), "David", "david@example.com");

            assertThat(updated.isSuccess()).isTrue();
            assertThat(updated.orElseThrow().name()).isEqualTo("david");
        }

        @Test
        @DisplayName("returns failure for unknown id")
        void returnsFailureForUnknownId() {
            var result = service.update(8888L, "Ghost", "ghost@x.com");

            assertThat(result.isFailure()).isTrue();
        }
    }

    @Nested
    @DisplayName("delete()")
    class Delete {

        @Test
        @DisplayName("removes an existing user")
        void removesExistingUser() {
            var user = service.create("Eve", "eve@x.com").orElseThrow();

            assertThat(service.delete(user.id())).isTrue();
            assertThat(service.findById(user.id())).isEmpty();
        }

        @Test
        @DisplayName("returns false for unknown id")
        void returnsFalseForUnknownId() {
            assertThat(service.delete(7777L)).isFalse();
        }
    }
}
```

## Step 6 — Compile and run the tests

Compile first to check everything fits together:

```bash
mvnd compile
```

If you see compilation errors, they are likely import mismatches between the generated package names. Check that the `import` statements in `UserService.java` match the actual package of `User` and `UserValidation`. Generated code is a starting point — adjust imports and package names to match your project structure.

Run the tests:

```bash
mvnd test -Dtest=UserServiceTest
```

All tests should pass. If any fail, read the error message — the generated tests document the expected behaviour clearly.

---

## Step 7 — Run jgen verify to confirm everything compiles

`jgen` ships with a built-in verify command that compiles, formats, and runs tests in one step:

```bash
bin/jgen verify
```

This is equivalent to `mvnd verify` but focused on the generated artefacts. Use it as a sanity check after any generation session.

## Step 8 — Explore the refactor pipeline

`jgen` can also analyse existing code and suggest migrations to modern Java 26 idioms:

```bash
# Analyse existing source and print a modernisation score + ranked suggestions
bin/jgen refactor --source ./src/main/java/com/example --score
```

Sample output:

```
Modernisation Report
====================
UserService.java        score=72  (opportunities: virtual-thread executor → Proc, raw Map → ConcurrentHashMap)
User.java               score=95  (well-modernised)
UserValidation.java     score=88  (opportunity: nested if-else → switch expression)

Top recommendations (by ROI):
  1. bin/jgen generate -t concurrency/virtual-threads -n UserService -p com.example.service
  2. bin/jgen generate -t core/pattern-matching       -n UserValidation -p com.example
```

Generate an executable migration plan:

```bash
bin/jgen refactor --source ./src/main/java/com/example --plan
# Writes: migrate.sh

bash migrate.sh   # applies all suggested migrations
```

| Command | Output |
|---|---|
